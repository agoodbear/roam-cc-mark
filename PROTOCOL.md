# CC 讀取／回寫協定（請CC修改）v2

> 定位：這是 Bear 的「紅筆批註」工具。**紙是 Bear 的，紅圈是 Bear 畫的；CC 只能在圈裡寫鉛筆字（提案），Bear 點頭才變墨水。**
> CC（Claude Code，透過 Roam MCP 從外部）改稿前**先讀完這份**。原稿範圍 = **一個 Roam page，一頁一篇**。

## 一、憲法（凌駕一切流程，違反即中止）

1. **CC 不得直接 `update_block` 任何原稿 block。**（原稿＝Bear 的字）
2. **CC 不得刪除 Bear 建的標記 block。**（清除只由 Bear 的 ✅ 觸發，透過 extension）
3. **CC 不得在原稿區新增 block。**
4. CC 的所有產出，只能寫進「**標記 block 本身（提案欄）**」——換句話說，CC 的字永遠關在標記裡、帶著身分，不匿名混入正文。
5. Bear 是唯一有權讓字進入原稿的人。

## 二、標記格式

Bear 建立（待處理）：
```
#請cc修改 【指令】<意圖>：<一句話> 【第N處】 【原文】「<精確片段>」
```
- 意圖 ∈ {潤, 接, 查, 議}（見四）。`【第N處】` 只在原文重複時出現，指第幾個出現。無【原文】= 整段。

CC 回寫（待審）＝把 tag 從 `#請cc修改` 換成 `#cc提案`，並補欄位，**欄位順序固定**：
```
#cc提案 【指令】…【第N處】【原文】「…」【提案】「<替換/草稿文>」【備註】<一句說明/來源>
```

## 三、一次讀全（單一 MCP call，別逐個撈）

待處理（Bear 剛丟給你的）：
```clojure
[:find ?childUid ?markStr ?parentUid ?parentStr
 :where
   [?t :node/title "請cc修改"]
   [?c :block/refs ?t] [?c :block/uid ?childUid] [?c :block/string ?markStr]
   [?p :block/children ?c] [?p :block/uid ?parentUid] [?p :block/string ?parentStr]
   [?p :block/page ?pg] [?pg :node/title "<頁名>"]]
```
把 `"請cc修改"` 換成 `"cc提案"` 即撈「已提案待審」的。Bear 也可在 extension 右下點 📋 打包成一段文字（含 page uid）貼給你，你就零查詢開工。

## 四、協作意圖（四種，各有明確回寫形態）

| 意圖 | Bear 要的 | CC 做什麼 | 回寫 | 守住 Bear 的聲音 |
|---|---|---|---|---|
| **潤** | 這句照我意思改好（口語化/縮短/去AI腔/台灣用語都算） | **最小編輯距離**改被圈的字 | `【提案】「替換文」`；tag→cc提案 | 只動被圈的字，保留 Bear 用詞、句式、判斷語氣；**禁擴寫禁加料、禁加 AI 過場語**；改幅超出指令寧可退回問 |
| **接** | 我卡住了，幫我起一段草稿 | 讀整頁上下文 + 風格檔 `reference_ecg_writing_style_profile` 寫草稿 | `【提案】「草稿」`；tag→cc提案 | 這是素材不是成品；被接受後會自動掛 `#cc草稿`＝還不是 Bear 的字，等 Bear 改寫收編 |
| **查** | 這數字/引用/說法可靠嗎、補來源 | 查證（**醫學數字照鐵律：兩獨立路徑交叉、走 OpenEvidence**） | `【備註】結果＋來源` **＋【提案】把來源自然織進原句的整合版**（＝原文＋citation，最小編輯、不重寫整句、不改語氣）；tag→cc提案 | 提案只把來源織進 Bear 原句；**查不實／不宜改字時不給【提案】**、只寫【備註】「待確認」，禁編造 |
| **議** | 給我 2–3 個說法選項／順序建議 | 給選項或結構建議 | `【備註】編號清單` **＋（若有明確首選）【提案】首選的改寫版**；tag→cc提案 | 提案＝你建議的首選寫法供一鍵套用；Bear 仍可「完成不改／退回」 |

回寫形態：**替換型**（潤，及查/議的整合版）、**插入型**（接）、**註記型**（查/議的【備註】）。**查/議＝【備註】(意見)＋【提案】(整合版) 並存**：Bear 在卡片上可按「✅ 套用整合版」(替換原句) 或「完成·不改」(只清標記)。整合版寫不出或不該寫就只留【備註】、不給【提案】。

## 五、對帳清單（回寫完，在 chat 回一份，Bear 一眼知道回 Roam 要看什麼）

一處一行：
```
#3 潤｜「到院前已去顫三次」→ 建議「到院前電擊了三次」（12字→10字）
#5 查｜D2B 90 分鐘 → 已證實，來源：2013 ACCF/AHA STEMI Guideline
#7 接｜（第 4 段結尾）已補一段 82 字草稿，接受後請改寫收編
```

## 六、生命週期與收斂

```
#請cc修改（Bear 建，待處理）
   → CC 回寫 → #cc提案（待審，含【提案】/【備註】）
      → Bear ✅ 接受 → extension 套用（潤=替換原文第N處／接=插入新 block 掛 #cc草稿／查議=「套用整合版」替換原句 或「完成·不改」只刪標記）
      → Bear ↩ 退回 → 改指令 → tag 換回 #請cc修改 → 再送（面板會顯示你上一版提案當參考）
      → Bear ✎ 改進這句（查/議專用）→ 轉成潤稿請求，指令已帶上你查到的來源/建議
         （例：潤：把查證到的來源整合進這句：<來源>）→ 再送 → CC 提「原文＋來源」整合好的句子
（標記消失＝這處收斂）  全頁標記＝0 ＝ 定稿

旁支（整篇重排版，見九）：
#cc排版提案（CC 回寫的頁面級提案 root，含【變更摘要】/【建議】/【重排結果】）
   → Bear ✅ 套用 → extension 原子三步（頁底建 🗄 備份 → 現有正文搬進備份 → 【重排結果】促升為正文、##/### 轉 heading、刪提案 root），套用前重驗歸零＋零位移
   → Bear ↩ 退回 → extension 刪整份提案（原稿本來就沒動過）
   套用後隨時可 ↺ 還原（把當前正文再存一份、促升回原稿）／🧹 清備份（confirm）
```

## 七、定稿轉 Hugo（三個歸零條件）

Bear 說「〈頁名〉轉 Hugo」時，CC 先檢查：**① 標記＝0 ② `#cc草稿`＝0**（未收編的 CC 草稿）**③ `#cc排版提案`＝0**（未處理的整篇重排版提案——見九；有提案代表版面還沒定，先讓 Bear 套用或退回）。
- 不滿足 → 列出擋下，不轉。
- **照片**：草稿裡 `![📷 …](https://composer.agoodbear.com/r/<refId>?s=…)` 是 Blog Composer 照片參照。成稿時抓出 refId → `POST localhost:8765/api/roam-ref-fetch` 換原檔 → 走 Hugo 媒材管線（詳見 blog-composer/ROAM-REFS.md）。那條 `/r/` URL 是縮圖不是原檔，別直接用。
- 滿足 → 讀整頁產 `content/posts/<type>-post-N.md`（沿用 ecg/study/travel/erlife-post-N 慣例；**Hugo 禁 H1**；排除「🗂 素材」子樹**與「🗄 …排版前備份／還原前狀態」`#cc排版備份` 子樹**；跑 zhtw-mcp lint）→ 部署 → 回 Roam 頁首寫一行 `✅ 已發佈 → <url> <日期>`，該頁視為封存（之後小修直接改 Hugo，大改才重開 Roam 頁）。
- 若有明示保留的 `#cc草稿` 段落 → 轉稿前必過 `/de-ai-zhtw`（只掃 CC 草稿出身的段落，Bear 原文絕不進）。

## 八、聲音守則（獨立成節，最高優先）

- **潤**：最小編輯距離、保留 Bear 用詞與判斷句、禁 AI 過場語與擴寫。
- **接**：先讀 `reference_ecg_writing_style_profile`；產出自知是草稿，不假裝成 Bear 的定稿。
- **查**：醫學數字兩路徑交叉、走 OpenEvidence；查不實寫「待確認」，禁編造引用/數字。**查證的【提案】整合版＝最小編輯把來源織進 Bear 原句（不重寫、不改語氣），寫不出/不宜就不給提案**。
- 一切以「這是 Bear 的文章、Bear 的聲音」為最高判準。CC 是紅筆，不是作者。

## 九、整篇重排版（📐 Reformat）

Bear 在 extension 右下點 **📐 重排版**（歸零才給打包）→ 複製一份「排版任務」貼給新開的 CC。這是**「排版」不是「改稿」**：CC 只出一份隔離的提案樹，extension 逐字機械驗證後、由 Bear 一鍵套用。

**憲法豁免（唯一一處）**：`#cc排版提案` root 是 §一.3「不得在原稿區新增 block」的**唯一豁免**——它是**頁面級的標記 block**，跟 `#請cc修改` 同性質，CC 的字仍然關在標記裡、帶著身分、不匿名混入正文。除了這棵提案樹，原稿一個 block 都不准動（不 update、不 delete、不 move）。

**鐵律（違反任一條＝任務作廢）：**
1. Bear 的每一個字逐字保留：禁改字、禁換詞、禁增刪內容、禁修錯字、禁加過場句。extension 會逐字機械比對＋格式記號守恆計數，有位移＝整份提案被鎖死、套不了。
2. 只能做六件事：① 加標題（獨立 block、`## `章節／`### `小節；用語取自 Bear 內文詞彙、短、像 Bear 口氣、禁 AI 腔標題）② 切分過長段落（只在原有標點處切，不增刪字元）③ 合併零碎段落（直接串接、不補字補標點；要補才通順→寫【建議】）④ 層級化（連續平行短句縮排為子層）⑤ 整句加粗（只對「可直接抄進筆記的臨床結論／判斷整句」加 `**…**`，全篇新增 ≤5 處，每處列進【變更摘要】）⑥ 清雜訊（刪純空白 block）。Bear 既有的 `**`、`^^`、`[[]]`、`(())`、`{{}}`、圖片連結原樣保留、不增不減。
3. 不碰：「🗂 素材／背景」子樹、「🗄」備份子樹、「✅ 已發佈」行、所有 `#標記` block。照片 block（`![📷 …](composer.agoodbear.com/…)`）逐字保留、跟著原本相鄰段落放。
4. 禁止段落搬移／跨節重排（敘事順序是 Bear 的作者判斷）。覺得順序該動→寫【建議】。
5. 你的全部產出只放進下述提案樹。

**提案樹格式**：在頁面**最底部**建 top-level block `#cc排版提案 【整篇重排版】YYYY-MM-DD`，其下三個子 block：
- `【變更摘要】標題 +N｜切分 N｜合併 N｜加粗 N｜清空行 N`——子層逐條列明細（每個新標題全文、每處合併/切分/加粗的位置與原文前 10 字）。
- `【建議】`——需改字才能解的排版問題，只建議不動手（沒有就寫「無」）。
- `【重排結果】`——其**直接子層＝重排後的完整正文樹**（每個頂層段落一個 block，標題 block 用 `##`／`###` 前綴，層級用縮排）。

**extension 端（Bear 按的、CC 不做）**：零位移驗證（去 `**`/`__`、壓空白後逐字串接比對＋格式記號次數守恆；剔除提案側 `##`/`###` 標題 block）→ 驗過才解鎖 ✅ 套用 → 原子三步（頁底建 `🗄 排版前備份 … #cc排版備份`(collapsed) → 現有正文搬進備份 → 【重排結果】促升為正文、`##`/`###` 轉 Roam heading 屬性、刪提案 root）。原稿永不刪（備份留著，Bear 手動 🧹 才清）。CC 只負責產出合格提案；套用/還原/清備份都是 Bear 在 extension 上的動作。
